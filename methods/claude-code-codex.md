# Claude Code & Codex

Claude Code and Codex are not the same thing as the standard ChatGPT interface or the regular Claude web chatbot. They are agentic research and coding assistants that can work across files, folders, prompts, notes, scripts, and outputs in a more structured way. That makes them especially useful for thesis work, where the challenge is rarely just answering one question. The real work is building a workflow you can defend, repeat, and explain.

<div class="info-box" markdown="1">
**Short version:** use chatbots for brainstorming or quick explanation; use Claude Code and Codex when you need help inspecting files, organizing research material, transforming data, writing or checking code, and documenting a process from start to finish.
</div>

## How they differ from chatbot interfaces

| | ChatGPT / Claude chat | Claude Code / Codex |
|---|---|---|
| **Typical interaction** | Ask a question, get a response | Work across a research folder, files, scripts, and tasks |
| **Memory of your project** | Mostly bounded to the chat thread | Can inspect and act on the actual project workspace |
| **Best for** | Brainstorming, quick explanations, drafting | Reproducible workflows, data cleanup, coding, structured research support |
| **Relationship to code** | Can discuss or suggest code | Can help build, revise, test, and document working pipelines |
| **Research value** | Useful but often conversational | Useful when the work itself is procedural, file-based, or iterative |

## Why they are powerful for thesis research

They can help you build a more organized and higher-quality workflow because they operate on the actual moving parts of a project rather than only on pasted text. In practice, that means they can:

- help structure folders, filenames, and metadata consistently
- turn a messy document collection into a clean and documented corpus
- generate or revise scripts for preprocessing, coding, OCR, parsing, or analysis
- check whether your coding decisions, categories, or file transformations are consistent
- help you keep notes, outputs, and methods aligned across drafts
- make it easier to rerun the same process instead of improvising it again later

The gain is not that they "do the thesis for you." The gain is that they can reduce avoidable disorder and help you work in a more explicit, inspectable way.

## Useful in qualitative work too

These tools are especially valuable for computational projects, but they are not limited to them. In qualitative or mixed-methods work they can still help you:

- organize archives, interview transcripts, policy documents, or media texts
- build coding frameworks or structured note-taking templates
- prepare documents for close reading or comparative analysis
- extract quotations, track themes, and maintain consistent evidence logs
- audit whether your references to cases, themes, or concepts stay internally consistent across a chapter

That means they can support discourse analysis, framing analysis, comparative case studies, process tracing, and literature-review work just as much as they support topic models or sentiment pipelines.

## Especially useful in computational and quantitative work

This is where Claude Code and Codex become particularly powerful. They can help you:

- preprocess text corpora
- clean OCR output and normalize document structure
- write scripts in Python or R
- move from raw files to analysis-ready tables or corpora
- document each transformation step so the workflow is reproducible
- troubleshoot errors in a script, environment, or pipeline

If your project involves the [Corpus Building Wizard](https://scdenney.github.io/corpus-building/), local scripting, or HPC workflows, these tools are often the most efficient way to move from a conceptual plan to an actual working pipeline.

## What responsible use looks like

Used well, these tools support better research. Used badly, they obscure what you actually did. The standard should be transparency and control.

- You remain responsible for the design, interpretation, and final claims.
- You should understand the workflow well enough to explain it to your supervisor.
- You should document meaningful prompts, scripts, and transformation steps.
- You should verify outputs rather than accepting them uncritically.
- You should disclose significant AI assistance when program rules require it.
- You should never present generated analysis, coding, or prose as if it emerged without assistance.

<div class="reflection-box" markdown="1">
<p>Good question to ask yourself</p>
<ul>
<li>Could I explain what the tool did, what I checked, and why I accepted the result?</li>
<li>If I had to rerun this process next month, would I know how?</li>
<li>Would my supervisor be able to understand the chain from source material to result?</li>
</ul>
</div>

## A good division of labor

Let the tool help with workflow, structure, cleanup, transformation, and iteration. Keep the human researcher responsible for judgment, interpretation, and argument.

That usually means:

- **Human decides** the research question, corpus boundaries, theory, method, and interpretation.
- **Tool assists** with preparation, organization, scripting, checking, and documentation.
- **Human verifies** that the evidence and claims still hold.
